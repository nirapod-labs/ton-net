# ton-net v0.2.0 implementation plan

The trust-minimization core. v0.1.0 read chain state and returned it as the
server's unproven word. v0.2.0 builds the machinery that lets a caller check that
word: the cell and bag-of-cells engine, the level-mask representation hashing that
gives every cell its identity, the TL-B for the account and block structures, and
the Merkle-proof routines that bind a liteserver's answer to a block hash the
caller already trusts.

After this milestone an account read is verified, not merely reported: given a
masterchain block hash the caller vouches for, the client proves the returned
account is the one in that block's state and rejects a tampered proof. It decodes
that account into a balance, a status, and the code and data cells. The one thing
still trusted is the block hash the caller supplied; removing that last trusted
input is block sync, the next milestone. Because the anchor is still caller-given,
a read is not yet trust-minimized end to end, so v0.2.0 ships as a git tag and, if
published, only as an `-alpha` pre-release (NET-ADR-004, the roadmap release model).

The riskiest piece, level-mask representation hashing, is proven against a real
mainnet account proof in [spikes/proof-feasibility](../../spikes/proof-feasibility)
before the library build starts, the same spike-first discipline the ADNL path
followed in v0.1.0.

## Delivers

From Rust and from Node: read an account at a masterchain block whose hash the
caller trusts, verify the liteserver's proof against that hash, and get the
account decoded into a balance, a status, and code and data cells. A tampered
proof is refused. An unverified, decoded read stays available for callers who do
not have a trusted hash yet.

What a consumer can do after v0.2.0:

- Parse a bag of cells into a cell tree, and serialize a cell tree back to a bag
  of cells.
- Compute the representation hash of any cell, ordinary or exotic, so a cell has a
  stable identity.
- Read an account and get its balance in nanotons, its status (active, frozen,
  uninitialized, or nonexistent), its last-transaction logical time, and its code
  and data cells when active.
- Verify that account against a caller-supplied trusted masterchain block hash: the
  client checks the block, shard, and state proofs and binds the returned account
  to that block, or returns an error.
- See a tampered proof rejected rather than silently accepted.

What a consumer cannot yet do, and the release says so: verify a read without
supplying a trusted block hash, because the sync anchor that produces one from a
single pinned key-block is v0.3.0; resolve an address through the DHT (v0.5.0); or
run a get-method such as a jetton balance or a DNS lookup, because that needs the
TVM (v0.7.0, NET-ADR-005). A verified balance is a proven read; a get-method result
is not proven until the TVM lands.

## Scope

In scope:

1. **Cell engine** in a new `ton-net-cell` crate: the cell model with ordinary and
   exotic cells (pruned branch, library reference, Merkle proof, Merkle update), the
   level mask, and level-mask-correct representation hashing; bag-of-cells parse and
   serialize; a slice reader for typed reads over a cell; a 256-bit dictionary lookup
   over the hashmap encoding, enough to find one key.
2. **Block and account TL-B** in a new `ton-net-block` crate: decode `Account` and
   its storage into a domain `Account`; decode the fragments of `Block`,
   `ShardStateUnsplit`, and `ShardAccount` that a proof walk touches.
3. **Proof engine** in `ton-net-block`: a Merkle-proof verification primitive and the
   `check_*` routines that compose it into a verified account read (block header
   proof, shard proof, account-in-state proof, and the account binding).
4. **Verified reads** in `ton-net-lite` and the `ton-net` facade: a `Verified<T>`
   wrapper beside the existing `ServerReported<T>`, an `account` that returns a
   decoded but unverified account, and an `account_verified` that returns a proven one.
5. **Node binding** update: the decoded account across the FFI boundary, amounts as
   decimal strings, and the verified read path.

Out of scope for v0.2.0, each with its own later milestone: **block sync** and the
pinned key-block anchor (v0.3.0), which is what removes the caller-supplied hash;
the **DHT** and ADNL-over-UDP (v0.5.0); the **TVM** and get-method execution
(v0.7.0); transaction and full block-list proofs beyond the account read path; a
general dictionary or hashmap library beyond the single-key lookup the account walk
needs; and decoding extra-currency balances (the grams balance is decoded, an
extra-currency dictionary is skipped, not parsed). Serialization produces a valid
bag of cells but not a byte-for-byte match of any particular encoder, because the
format admits several valid encodings; round-trip and hash equality are the
correctness measures, not byte identity.

## Workspace layout

Two crates join the workspace, on the durable seams the system design already
names: a cell primitive below everything TON-specific, and a block-model-and-proof
crate above it. The crates move in lockstep (NET-ADR-004); the split is for
boundaries, not independent versioning.

```
ton-net/
  Cargo.toml                  workspace
  crates/
    ton-net-tl/               TL codec and the schema types this release reads
    ton-net-cell/             NEW: cell model, exotic cells, level-mask hashing,
                              BoC parse and serialize, slice reader, dict lookup
    ton-net-adnl/             ADNL transport, handshake, framing, driver
    ton-net-block/            NEW: block and account TL-B, the proof engine
    ton-net-lite/             liteserver read client, verified reads
    ton-net/                  facade: Config, Address, Client, Verified, Error
  bindings/
    node/                     napi-rs binding
  spikes/
    adnl-feasibility/         the v0.1.0 proving spike, kept as evidence
    proof-feasibility/        NEW: the level-mask hashing and proof proving spike
  docs/
```

Dependency direction stays one way and shallow. `cell` depends on nothing in the
workspace. `block` depends on `cell`. `lite` depends on `block`, `adnl`, and `tl`.
The `ton-net` facade depends on `lite`. The Node binding depends on the facade. No
cycles, and the facade is still the only crate a consumer names. The `cell` and
`block` crates decode from cells by hand and do not touch `tl-proto`; the bag of
cells is a distinct encoding from the TL wire format, below it in the stack.

## The trust model: reported, verified, and the anchor

v0.1.0 gave every read one status: reported, the server's unproven word, marked in
the type as `ServerReported<T>`. v0.2.0 adds a second status and one input.

- **Reported** (`ServerReported<T>`, unchanged): the liteserver's answer, decoded
  but not proof-checked. Available without a trusted hash.
- **Verified** (`Verified<T>`, new): an answer whose Merkle proof was checked to
  root at a block hash the caller supplied, and whose account was bound to that
  block's state. The wrapper carries the anchor it was checked against, so the
  provenance is explicit: this value is proven to sit in the state of that block.
- **The anchor**: a masterchain `BlockIdExt` the caller trusts the root hash of.
  In v0.2.0 the caller obtains it out of band, from a source they already trust.
  The client reads the account at exactly that block and proves the answer against
  it. It never invents or upgrades the anchor.

This is the honest shape of a proof engine without sync. Verification here reduces
trust to a single input, the anchor, rather than removing it. The registry channel
matches the code: `-alpha` only, because "trust one hash the caller brought" is not
yet "trust nothing but a pinned key-block." v0.3.0 produces the anchor from sync and
becomes the first ordinary release (NET-ADR-004).

## Architecture: the proof engine is a pure function

NET-ADR-001 fixes the core as sans-I/O. v0.2.0 keeps the whole new surface pure and
synchronous. Nothing added here touches a socket.

- **The cell engine is pure data.** Parsing a bag of cells, hashing a cell, reading
  a slice, and serializing are functions over bytes. They are fully testable with no
  network and no runtime.
- **The proof engine is a pure function** of three inputs: the trusted anchor hash,
  the proof bytes the server sent, and the account state bytes. It returns a verified,
  decoded account or an error. Given the same inputs it gives the same answer, so a
  fixed proof captured once becomes a hermetic regression test that never reaches the
  network.
- **The async edge is unchanged.** `AdnlConnection`, `LiteClient`, and the facade
  `Client` keep the v0.1.0 shape. The one new async method, `account_verified`, reads
  at the anchor block and then calls the pure proof engine; the verification itself is
  synchronous.

This is why the milestone can be proven mostly off the network: the hard part, the
hashing and the proof walk, is deterministic over bytes, and the mainnet corpus is
there to confirm the bytes were captured from something real.

## The cell engine and level-mask hashing

The cell is TON's universal container: up to 1023 bits of data and up to four
references to other cells, forming a directed acyclic graph. A cell has a type,
ordinary or one of four exotic kinds, and a level mask that governs how many hashes
it carries. The representation hash of a cell is its identity, and a Merkle proof is
believed exactly because recomputing those hashes reproduces a known root. Getting
the hashing right is the whole game; the roadmap flags it as the subtle, critical
piece, and this plan treats it as such.

**Cells.** An ordinary cell is data bits plus child references. Its level is the
maximum of its children's levels, and it carries one representation hash at that
level plus a depth. The two descriptor bytes that head a cell encode the reference
count, the exotic flag, and the level mask (the first) and the bit length (the
second).

**Exotic cells.** Four kinds, each identified by its first data byte when the exotic
flag is set:

- **Pruned branch**: stands in for an omitted subtree in a proof, storing that
  subtree's hash and depth so the parent still hashes to the same value. This is what
  makes a proof small: everything off the proved path is a pruned branch.
- **Merkle proof**: one reference, storing the claimed hash and depth of that
  reference. Verifying it means recomputing the reference's hash and checking it
  equals the stored claim; the reference, virtualized, is the proved tree.
- **Merkle update**: two references, an old and a new subtree, each with a stored
  hash and depth. A block's state update is one of these.
- **Library reference**: names contract code by hash. Recognized and represented; not
  resolved in this milestone.

**Representation hashing.** The hash is a SHA-256 over the cell's standard
representation: the two descriptor bytes, the data padded to a byte boundary, then
each reference's depth as two big-endian bytes, then each reference's 32-byte hash.
A cell with a nonzero level mask carries several hashes and depths, one per set level
plus level zero, and the higher hashes substitute the corresponding higher hashes of
the children. Pruned branches carry their higher hashes in their data rather than
recomputing them; Merkle cells lower the level of their content by one. This
level-mask rule is the exact spot the roadmap warns pytoniq gets subtly wrong, so it
is implemented from the specification and pinned by vectors, not copied.

**How it is de-risked.** Three layers of check, strongest first:

- A **mainnet root match**: the spike takes a real account proof at a block whose
  state hash comes from an independent source, recomputes the Merkle root through the
  level-mask hasher, and requires it to equal that hash. A wrong hashing rule cannot
  pass this, because the root is a SHA-256 over the whole tree.
- **Known-answer vectors**: the empty cell hash
  (`96a296d2...0a09cfc7`), a handful of small cells and trees whose hashes come from an
  independent TON tool, and a captured pruned branch, all checked in the hermetic run.
- **Round-trip**: parse then serialize then parse again reproduces the same cells and
  the same hashes.

**Refusing hostile bytes.** A liteserver is not trusted, and a proof is
attacker-shaped by design, so the parser and the hasher must not be a denial-of-service
surface. The cell engine bounds cell count and tree depth, rejects a reference that
does not point strictly forward in the bag (which also forbids cycles), and rejects a
declared length that does not match the bytes. Parsing any byte string terminates and
yields cells or a `Cell` error, never a panic or an unbounded allocation; a fuzz target
guards this, as the TL decoder's does.

## The proof engine

The proof engine lives in `ton-net-block` and composes one primitive into a verified
account read. Everything below is a pure function of bytes and the anchor.

**The primitive.** `verify_merkle_proof(proof_cell, expected_hash) -> VirtualCell`:
recompute the hash of the Merkle-proof cell's content and require it to equal
`expected_hash`; return the content as a virtual cell in which pruned branches are
opaque leaves of known hash. Everything else is a walk over virtual cells.

**The account read, verified.** Given the anchor block hash, the `shard_proof`,
`proof`, and `state` a liteserver returned for `getAccountState`, and the account id:

1. **Block header proof**: verify a Merkle proof rooted at the anchor block hash that
   exposes the block's info and state update, and read the masterchain state hash from
   the update's new-hash.
2. **Shard proof** (basechain only): from the masterchain state, verify the account's
   shard block is the one recorded, and read its state hash. For a masterchain account
   this step is empty and the account lives in the masterchain state directly.
3. **Account-in-state proof**: verify a Merkle proof rooted at the relevant state hash
   that exposes the accounts dictionary, and look the account id up in it to reach the
   `ShardAccount` leaf, which carries the account cell's hash (pruned in the proof).
4. **Binding**: parse the returned `state` as a cell, hash it, and require that hash to
   equal the account hash the dictionary lookup produced. Only now is the state known to
   be this account at this block.
5. **Decode**: decode the bound account cell into the domain `Account`.

A failure at any step is `Error::Proof`, never a partial or a guessed result. A
`state` that does not exist at the block is a proven-nonexistent account with a zero
balance, not an error.

**The five routines.** The primitive plus steps 1 through 4 are the `check_*_proof`
set the roadmap names for this milestone: the Merkle-proof primitive, the block header
proof, the shard proof, the account-in-state proof, and the binding. The block-to-block
link proof that walks key-blocks is deliberately not here; it is the sync engine, v0.3.0.

**Masterchain first, then basechain.** The masterchain account path (steps 1, 3, 4, 5)
is simpler and is proven first, on the always-active elector and config contracts. The
basechain path adds the shard proof (step 2) and is proven second, so both workchains
are covered by the gate.

## Error taxonomy

The facade keeps its single `#[non_exhaustive]` `Error`, with two variants added.
The lower crates carry their own narrow errors that the facade maps in: a
`CellError` from `ton-net-cell`, a `BlockError` from `ton-net-block`.

```rust
#[derive(Debug, thiserror::Error)]
#[non_exhaustive]
pub enum Error {
    // ... the v0.1.0 variants, unchanged ...

    /// A cell or bag of cells could not be decoded.
    #[error("cell: {0}")]
    Cell(String),

    /// A proof did not verify, or the account did not bind to the trusted block.
    #[error("proof: {0}")]
    Proof(String),
}
```

Two invariants hold, both testable and both new here:

- **Verification is sound in the negative.** A proof that does not root at the anchor
  hash, or an account that does not bind, yields `Error::Proof`, never a value. A
  corpus of tampered proofs, each a real proof with one byte flipped, is rejected.
- **Parsing hostile bytes terminates.** Any byte string handed to the cell parser or
  the proof engine yields cells, a value, or an error, never a panic or an unbounded
  allocation. A fuzz target guards the cell parser.

## Numbers across the FFI boundary

v0.1.0 fixed the mapping and deferred the coin case to here. v0.2.0 realizes it.

- **Balances are decimal strings.** A nanoton balance is a `u128`; a JS number holds
  it only to `2^53`, so it crosses as a decimal string, for example `"1234567890"`.
  This is the promise the v0.1.0 plan recorded, now exercised.
- **Logical time is a string.** `last_trans_lt` is a `u64` that already exceeds
  `2^53` on mainnet, so it crosses as a decimal string, not a number.
- **Status is a tag with fields.** The status crosses as a string, `"active"`,
  `"frozen"`, `"uninit"`, or `"nonexistent"`, with `code` and `data` present as
  buffers only when active, and `stateHash` present only when frozen.
- **Code and data are bags of cells.** Each crosses as a `Buffer` holding the cell
  serialized as a single-root bag of cells, the interchange form other TON tools read.
- **The anchor stays structured.** A verified read carries its anchor block id, with
  its `u64` shard as a hex string, as in v0.1.0.

## The Node binding surface

The TypeScript a consumer sees, generated by napi-rs from the binding:

```ts
export type AccountStatus = "active" | "frozen" | "uninit" | "nonexistent"

export interface Account {
  balance: string        // nanotons, decimal
  status: AccountStatus
  lastTransLt: string    // u64, decimal
  code?: Buffer          // BoC, present when active
  data?: Buffer          // BoC, present when active
  stateHash?: Buffer     // present when frozen
}

export interface BlockId {
  workchain: number
  shard: string          // u64 as hex
  seqno: number
  rootHash: Buffer
  fileHash: Buffer
}

/** A value a liteserver reported without a checked proof. Not verified state. */
export interface ServerReported<T> { value: T; proof: Buffer }

/** A value proven against a block the caller trusts. */
export interface Verified<T> { value: T; anchor: BlockId }

export class TonClient {
  static connect(config: Config): Promise<TonClient>
  masterchainInfo(): Promise<ServerReported<MasterchainInfo>>
  account(address: string): Promise<ServerReported<Account>>
  accountVerified(address: string, trusted: BlockId): Promise<Verified<Account>>
}
```

The two wrappers carry into TypeScript on purpose: a caller sees, in the type, whether
a value is the server's word or proven against an anchor.

## Documentation standard

Unchanged from v0.1.0, and it is a CI gate, not a preference. The two new crates set
`#![forbid(unsafe_code)]` and `#![warn(missing_docs)]` (denied in CI) and
`#![deny(rustdoc::broken_intra_doc_links)]`. Every public item is documented; every
entry point has a compiling doctest; every `Result` documents its `# Errors`; the cell
and proof functions document that they do not panic, which is a tested invariant. MSRV
for both new crates is the core floor, 1.85, checked in CI.

## Testing and conformance

Three layers, matching [docs/conformance.md](../conformance.md) for the families this
milestone touches (cells, hashing, and proofs).

- **Hermetic unit tests**, no network, on every commit:
  - Cells: parse and serialize round-trip on pinned vectors; the empty-cell hash and a
    small corpus of cell and tree hashes against an independent TON tool; a captured
    pruned branch and Merkle-proof cell hashed correctly; the fuzz target that must not
    panic or over-allocate.
  - Decode: a captured real account cell decoded to an exact balance, status, code, and
    data; the frozen, uninitialized, and nonexistent cases from hand-built cells.
  - Proofs: a captured real account proof verified against its known block hash; a
    corpus of tampered proofs, each rejected; the account binding, and a mismatched
    state rejected.
- **Network integration tests**, `#[ignore]`, in the dedicated network CI job: read and
  verify the elector and config contracts (masterchain) and a basechain account against
  a trusted hash obtained from an independent source; assert the decoded balance matches
  that independent source to the nanoton, at the same block, so the match is not racy;
  assert an unverified `account` decodes to the same balance.
- **The Node gate test**: from Node, `accountVerified` for a known account against a
  trusted block, asserting the decoded balance as a string, the status, and that a
  tampered proof rejects, all across resolved Promises.

The independent source is a public API separate from the liteserver being queried, so a
match is agreement between two implementations, not a server agreeing with itself. This
is the same cross-oracle discipline the v0.1.0 Node gate used for the masterchain seqno,
raised to a decoded balance and a checked proof. The cell-hash and proof vectors are
anchored to TON, generated against the reference node and an independent tool, and
checked in the hermetic run so they measure this implementation against TON, not against
itself.

## Dependencies

The v0.2.0 additions to NET-ADR-003, per crate. No new third-party crate is added
beyond what the ADR already lists; the cell engine reuses the SHA-256 already in the
tree.

- **ton-net-cell**: `sha2` (0.10, the representation hash), held at tl-proto's crypto
  generation as the ADR requires, a single copy in the tree; `thiserror`. The bag-of-cells
  CRC32C checksum is hand-written, as the address CRC16 was in v0.1.0, rather than adding
  a dependency. No `tl-proto`: the bag of cells is not the TL wire format.
- **ton-net-block**: `ton-net-cell` and `thiserror`. It decodes block and account
  structures by hand over the slice reader and holds the proof engine. It does not depend
  on `ton-net-lite`, so the dependency direction stays acyclic.
- **ton-net-lite** and **ton-net**: a new dependency on `ton-net-block`; no new
  third-party crate.
- **bindings/node**: unchanged dependency set.

The crypto tree stays single-copy: `cargo tree -d` shows one `sha2`, one `digest`, one
`crypto-common`, and one `block-buffer`, as NET-ADR-003 requires, and the cell hash uses
that one copy.

## Sequence

Ordered steps, each ending in something checkable. A step does not start until the one
before it is green.

0. **Feasibility spike** (`spikes/proof-feasibility`). A throwaway that captures a real
   `getAccountState` response for a known account at a masterchain block, sources that
   block's state hash independently, and recomputes the Merkle root through a first-cut
   level-mask hasher to confirm it matches. It also pins the account cell layout,
   including whichever `StorageInfo` variant mainnet currently uses, against the balance
   an independent source reports. Exit: the hashing rule and the account read proof are
   confirmed against mainnet, and the byte layouts are pinned, before the library build
   starts.
1. **ton-net-cell, parse and read.** The cell model with ordinary and exotic cells and
   the level mask; bag-of-cells parse with the bounds and rejections; the slice reader
   (bits, unsigned integers, coins, references, maybe, address). Exit: pinned and captured
   bags of cells parse to the right structure, malformed inputs are rejected, and the
   slice reads pinned values.
2. **ton-net-cell, hash and serialize.** Level-mask representation hashing for ordinary
   and exotic cells; cell virtualization; bag-of-cells serialize. Exit: the empty-cell hash
   and the vector corpus pass, a captured Merkle proof roots to its pinned block hash,
   and serialize then parse round-trips to the same hashes.
3. **ton-net-block, decode.** The account TL-B into the domain `Account`, and the
   fragments of `Block`, `ShardStateUnsplit`, and `ShardAccount` a proof walk reads; the
   256-bit dictionary lookup. Exit: a captured account decodes to the exact balance,
   status, code, and data, and the dictionary lookup finds an account in a captured state
   proof.
4. **ton-net-block, the proof engine.** The Merkle-proof primitive and the block, shard,
   account, and binding routines. Exit: a real account read verifies against an
   independently-sourced trusted hash for a masterchain and a basechain account, and a
   corpus of tampered proofs is rejected, all hermetic over captured bytes.
5. **ton-net-lite and the facade.** `Verified<T>`; retain `shard_proof` and `shardblk`
   in the lite read; `account` returns a decoded `ServerReported<Account>`;
   `account_verified` returns `Verified<Account>`. Exit: the hermetic suite is green and
   an ignored network test reads and verifies the elector against an independent hash with
   the balance matching an independent source.
6. **bindings/node.** The decoded account across the FFI boundary, amounts as decimal
   strings, and `accountVerified`. Exit: the Node gate test verifies an account, matches
   the balance string against an independent source, and rejects a tampered proof.
7. **Release.** Tag `v0.2.0`. If published, publish the Rust crates to crates.io and the
   Node binding to npm as `0.2.0-alpha.1` on the `alpha` channel (NET-ADR-004), still a
   pre-release because the anchor is caller-supplied.

## Decisions made in this plan

- **A cell crate below a block crate.** The cell model and bag of cells are a primitive
  with no TON-block knowledge, and the block model and proofs sit above it. Two crates
  keep each seam a single job, as the rest of the workspace does.
- **Verification reduces trust to an anchor, it does not remove it.** v0.2.0 proves an
  account against a caller-supplied block hash. This is the honest limit without sync, and
  it is why the release stays `-alpha`. The `Verified<T>` wrapper names the anchor so the
  reduction is legible, not hidden.
- **Level-mask hashing is implemented from the specification and pinned by a mainnet root
  match.** The roadmap flags the reference implementations as subtly wrong here, so the
  rule is not copied; it is written from the spec and proven by recomputing a real root.
- **The grams balance is decoded; extra currencies are skipped.** The nanoton balance is
  what a read needs; an extra-currency dictionary is stepped over so the parse continues,
  not decoded. Extra-currency decode is deferred.
- **Amounts cross the FFI boundary as decimal strings.** Fixed in the v0.1.0 plan, applied
  here to balances and logical time, because a JS number cannot hold either without loss.
- **The unverified read stays.** `account` still returns a decoded `ServerReported`, so a
  caller without a trusted hash is served, and the type still says the value is unproven.

## Open questions

- **The exact level-mask higher-hash rule.** Resolved in step 0 and pinned in step 2. The
  rule is implemented from the specification and confirmed by a mainnet root match and a
  vector corpus from an independent tool, so a subtle error cannot pass.
- **The current `StorageInfo` layout.** The account storage layout has changed across TON
  versions. It is pinned in step 0 against a real account whose balance an independent
  source reports, rather than asserted from memory.
- **`ton-net-block` versus folding into `ton-net-lite`.** Decided as a separate crate for
  the clean seam and the acyclic dependency direction. Revisit only if the block model
  stays small enough that a separate crate is overhead, which the proof engine makes
  unlikely.
- **The `Verified<T>` shape.** Decided to carry the anchor block id. An alternative that
  carries the full trusted proof for re-checking is heavier and not needed until a caller
  wants to re-verify offline; deferred.
- **Whether to record a proof-engine ADR.** The level-mask-from-specification decision and
  the anchor trust model are design choices worth a short ADR if they harden. Noted, not
  required for the plan.

## Definition of done

v0.2.0 is done when:

- From Rust and from Node, an account is read at a caller-supplied trusted masterchain
  block, its block, shard, and state proofs are checked, it is bound to that block, and it
  is decoded into a balance, status, and code and data. A masterchain account and a
  basechain account both pass.
- A tampered proof is rejected, proven by a corpus of one-byte-mutated real proofs.
- The decoded balance matches an independent public source to the nanoton, at the same
  block, for a corpus of real accounts.
- The cell-hash and proof vectors, the round-trip and decode tests, `fmt`, `clippy` with
  warnings denied, and the doc gate are all green, and the fuzz target runs clean.
- The network integration tests pass in their dedicated CI job, and the Node gate test
  verifies an account and rejects a tampered proof across the FFI boundary.
- `v0.2.0` is tagged. Any registry publish is `0.2.0-alpha.1` on the `alpha` channel,
  marked not for production, because the anchor is still caller-supplied.
