# ton-net v0.4.0 implementation plan

v0.4.0 brings the cell engine to full capability. It is the current milestone, and
NET-ADR-001 names that milestone the minimum coherent state the project holds itself
above: the cell engine with every gate green, on the way to the write path that backs
a wallet end to end. NET-ADR-003 places the cell engine at full capability inside the
client surface, builders, slices, the dictionary variants, augmented dictionaries,
usage trees, virtualization, and Merkle proof creation, so the work here is committed
scope rather than a capability sorted in by one consumer.

The engine lives in `ton-net-cell`, the base crate that depends on no other workspace
crate (NET-ADR-009). A cell reaches a caller from parsing or from a builder and from
nowhere else, and the crate forbids unsafe code and denies the panic and indexing lints
a decoder must not trip, because it reads bytes from a peer the client does not trust
(NET-ADR-003).

## The gate

A capability in this milestone is done when it is correct, safe, and performant
together. That ordering is fixed by NET-ADR-001: correctness and safety are the floor
and win any real conflict, and the floor is not permission to be slow. A capability that
is correct and safe but needlessly slow is unfinished.

Two oracles decide correctness, and both hold the engine to values the network produced
rather than to the engine's own arithmetic:

- A Merkle proof this crate builds verifies in the engine. A proof built over the tree
  a liteserver's proof stands for reproduces that server's proof cell, and virtualizing
  a captured proof reads back a tree that hashes to the root the proof claims. These run
  against a real account proof and a whole basechain block in
  `crates/ton-net-cell/tests/cell/merkle.rs`.
- Every dictionary variant rebuilds to its own mainnet root hash. The accounts
  dictionary of a whole mainnet block, built one entry at a time, built in bulk, and
  recombined from two halves, each reaches the root hash the validators computed for it,
  and removing what was added restores that tree with its summaries. These run in
  `crates/ton-net-cell/tests/cell/augmented.rs`.

The performance bar of NET-ADR-001 governs the same gate. The node and indexer scale
operations are in scope in this milestone and built now, not deferred. Their cost is
measured by criterion benchmarks on the hot paths (`crates/ton-net-cell/benches/`), and
each scale path is held to agree with the in-memory path while bounding the memory it
holds. Scale breadth is carried as isolated paths a lighter consumer does not compile
(NET-ADR-009), so the bar and the mobile binary size are held at once.

## Done

The capabilities below have landed against the two oracles.

### Builder and slice

`Builder` and `Slice` are the write and read cursors over a cell's bits and references,
with the address and snake-string specializations beside each trunk
(`crates/ton-net-cell/src/builder.rs`, `builder/address.rs`, `builder/snake.rs`,
`crates/ton-net-cell/src/slice.rs`, `slice/address.rs`, `slice/compare.rs`,
`slice/snake.rs`). `MsgAddress` reads and writes a message address in the forms the
chain uses. Named edges remain and are listed under Pending.

### The dictionary variants

The plain `HashmapE`, the augmented `HashmapAug`, and the prefix dictionary each carry
the full write path, get, set, remove, and iteration, over one descent, split, and
rebuild written once behind a private shape seam in the trunk
(`crates/ton-net-cell/src/dict.rs`) and shared by the shapes
(`dict/plain.rs`, `dict/aug.rs`, `dict/prefix.rs`, `dict/label.rs`, `dict/floor.rs`).

- Plain `Dict`: `new`, `get`, `set`, `remove`, `iter`, `count`, `min`, `max`,
  `take_min`, `take_max`, `retain`, `from_items`, the nearest-key `entry_at_or_after`
  and `entry_at_or_before`, and the prefix carve `subdict`.
- `AugDict`: `new`, `get`, `set`, `remove`, `iter`, `from_items`, `count`. A node
  carries a summary of the subtree below it, a fork's is the combination of its two
  children's, the meaning of a summary comes from the caller through the `Augmentation`
  trait, and a summary is recomputed from the two children on every write rather than
  carried forward. A write that would have to summarise a pruned branch is refused
  rather than guessed at.
- `PfxDict`: `new`, `get`, `set`, `remove`, `iter`.

The mainnet-root-hash oracle covers all three. A dictionary rebuilt from its own entries
in different orders reaches the identical canonical root, and that root is the one the
validators computed for the block the entries came from
(`crates/ton-net-cell/tests/cell/augmented.rs`,
`crates/ton-net-cell/tests/cell/whole_block.rs`).

### Usage tree and virtualization

A `UsageTree` records the cells a read touches, and `prune` rebuilds the tree with every
untouched subtree replaced by a pruned branch carrying only the hash and depth it stood
for (`crates/ton-net-cell/src/usage.rs`, `usage/trace.rs`). `virtualize` reads the tree a
Merkle proof stands for, with `is_virtualized` and `rebuild_with_refs` beside it
(`crates/ton-net-cell/src/merkle/virtualize.rs`). A mainnet block pruned to a single path
still hashes to the root it was served under (`tests/cell/merkle.rs`).

### Merkle proof creation and Merkle update

`create_proof` builds a proof over a single tree, and `create_update`, `apply_update`,
`combine_updates`, `may_apply`, and `validate_update` build and check the paired old-and-
new update (`crates/ton-net-cell/src/merkle/proof.rs`, `merkle/update.rs`, trunk
`merkle.rs`). A proof and an update built here reproduce a server's proof cell and a
mainnet block's state update cell (`tests/cell/merkle.rs`). A running virtualization
offset for proofs nested inside proofs is not built, because the structures this client
reads do not nest one; it is added when a structure that needs it does.

### The bag-of-cells variants

The read path is `parse`, the write path is `serialize`, and a `view` reads a bag
without building it. All three reach one header, one bounded reader, and one checksum in
the `boc.rs` trunk, so a bound is defined once and cannot hold in one path and lapse in
another (NET-ADR-009).

- Multi-root bags. A bag carries the indices of the roots it is read from; a liteserver's
  account proof has two, and both come back (`boc/serialize.rs`, `boc/parse.rs`,
  `tests/cell/mainnet_boc.rs`).
- Serialization options. `BocOptions` chooses the trailing CRC-32C checksum and the
  per-cell offset index, and every combination round-trips to the same cell identities
  (`boc/serialize.rs`).
- File hash. `file_hash` is the SHA-256 of a serialized bag, the second of the pair of
  hashes a block is named by (`boc/serialize.rs`).
- A produced-hash index. Shared cells are stored once, keyed by representation hash, and
  the optional offset index lets a reader locate a cell without walking the bag
  (`boc/serialize.rs`, header in `boc/header.rs`, reported by `BocView::has_index`).
- A read-only view. `BocView::open` reads and checks the header, cheaply and building no
  cells, and reports the counts and flags before `materialize` builds the graph
  (`boc/view.rs`).
- A random-access view that hash-verifies without materializing. `BocView::verify` runs
  every check `materialize` runs but keeps a summary of each cell rather than the cell,
  so a bag larger than its materialized graph is verified and its root hashes read, and
  `BocView::cell` builds one cell and only the subtree it reaches
  (`boc/random.rs`, `boc/parse.rs`).
- A per-reference lazy reader. `LazyBoc` builds the cell at an index on demand and keeps
  each cell it builds, so a caller that needs part of a large bag pays for that part and
  a later read of the same cell is free. The kept cells sit in the reader, so a `Cell`
  stays the immutable value it is everywhere (`boc/lazy.rs`).
- A one-pass streaming serializer. `serialize_boc_chunks` yields the same bytes
  `serialize_boc` produces, a chunk at a time, so a bag is written to a file or a socket
  without a second copy in memory. It matches the one-buffer serializer byte for byte on
  a bag the node itself produced (`boc/large.rs`, `tests/cell/mainnet_boc.rs`).

The scale readers and the streaming writer sit as isolated children of the `boc/` trunk
that the default read-and-build path a wallet takes does not compile or run
(NET-ADR-009).

### Cell dump and JSON

A cell renders as text for reading and logging, in the hex and binary notations TON's own
tools read (`crates/ton-net-cell/src/cell/dump.rs`). A JSON form of an ordinary cell tree
sits behind the `json` feature, off the default build, and round-trips: reading back what
it wrote rebuilds a cell with the same hash (`cell/json.rs`). The feature admits no new
dependency, only a new use of the `serde_json` already in the tree (NET-ADR-004).

## Pending

The work below remains before v0.4.0 ships. Each item inherits the gate above.

### Structure-aware BoC compression, self-contained

Today the only compression is plain LZ4 behind the `compress` feature, on the untrusted
decode boundary NET-ADR-010 fixes: a compressed bag names the length it expands to,
`decompress` refuses a length past a hard cap before allocating, and the expansion runs
on the bounds-checked safe-decode path (`crates/ton-net-cell/src/boc/compress.rs`). A
structure-aware form encodes a bag against the shape of a cell graph rather than as an
opaque byte stream, and it is built self-contained: this crate defines and reads its own
form rather than conforming to an outside codec.

The gate is a compress-then-decompress round-trip on a captured bag, and beating plain
LZ4 on real mainnet bags. It stays behind a feature, off the default build, and its
decode side inherits the boundary rules of NET-ADR-010: a cap before allocation, a
bounds-checked expansion, and a typed error rather than a panic or an unbounded
allocation on hostile input, with the single-copy crypto tree of NET-ADR-004 left intact.

### State-differential BoC compression

A differential form encodes a bag as the change from a base bag the two ends share, for
the case where a consumer already holds most of what a new bag carries. It is built
self-contained, behind the same feature and under the same boundary rules as the
structure-aware form, and held to the same round-trip gate: a differential compress
against a base, then a decompress against that base, reproduces the bag's cell
identities.

### The compression measured conclusion

The two forms are pursued to a measured outcome. If honest effort shows no safe,
self-contained form beats plain LZ4 on the hash-heavy bags this client reads, that
outcome is documented and plain LZ4 is kept as the shipped compression. That is a
measured conclusion, not a scope cut: the milestone reads and writes TON's compressed
bag either way, and the record states which form ships and the evidence for it.

### Remaining builder and slice edges

`Builder` and `Slice` carry the common write and read path. The remaining edges are the
less-used stores and loads and the boundary cases they surface, brought to the same
coverage the rest of the cursor holds
(`crates/ton-net-cell/src/builder.rs`, `slice.rs`).

### Serialize completeness, including the cached-hash form

The parser already reads the cached-hash form: a cell may carry its own hashes and depths
ahead of its data, one of each per level its mask marks and one more besides, and the
parser reads them and checks them against what the cell's contents give rather than
trusting them (`crates/ton-net-cell/src/boc/parse.rs`, the `d1 & 16` read and
`check_stored`). The serializer writes only the descriptors and the data and never the
stored hashes (`boc/serialize.rs`), and `BocOptions` carries no cached-hash choice. The
edge is to write the form the parser already accepts, so a bag the engine emits can carry
its cells' computed hashes for a reader that wants them without recomputation. The gate is
a round-trip: a bag serialized with cached hashes parses back, and its cells' stored
hashes match the ones their contents give.

### Proof, update, and metadata edges

Proof creation and the paired update cover the forms this client reads today. The
remaining edges are the shapes a fuller surface meets, and the cell `Metadata` and
`RefMetadata` accessors brought to the same completeness, held to the mainnet oracle that
already checks reported hashes and depths against the ones the engine computed
(`crates/ton-net-cell/src/merkle/`, `cell/metadata.rs`, `tests/cell/mainnet_boc.rs`).

### Base64 and hex transport utilities

A cell hash, an address, and a serialized bag travel as base64 or hex in the surfaces a
client speaks. The base64 handling sits today in the facade's address and config paths
(`crates/ton-net/src/address.rs`, `config.rs`), and hex rendering sits in the cell dump
and JSON forms. The edge is a transport codec the cell engine exposes for a bag and a
hash, canonical on the way in as the address decode already requires, so a caller encodes
and decodes without reaching for an outside crate.

### Wave-parallel finalization of a large bag

Finalizing a large bag, hashing every cell bottom up, is today a single pass. No
parallelism runs anywhere in the workspace. The scale edge is to finalize a large bag in
waves by graph depth, hashing the cells at one depth together before the depth above them
depends on them, so a many-cell bag an indexer builds finalizes faster while every cell
still hashes to the identity the single pass gives. It is a node-scale path, isolated so a
lighter consumer does not compile it (NET-ADR-009), and any dependency it needs enters
under the vetting and single-copy rules of NET-ADR-004. The gate is byte-for-byte
agreement with the single-pass finalization on real mainnet bags, and a measured speedup
on the large-bag benchmark.

### Surface and release

The facade re-exports the cell engine's public surface a consumer takes through
`ton-net`. Today it re-exports `Cell` and `CellType` (`crates/ton-net/src/lib.rs`); the
release surfaces the fuller engine a consumer of the facade uses. The step then bumps the
workspace version from 0.3.0 to 0.4.0 (`Cargo.toml`, `[workspace.package]`), records the
milestone in `CHANGELOG.md`, tags the release, and publishes. The Rust crates move in
lockstep on one library version, and the version is measured against the observable API
and the wire behaviour. This step is operator-gated.

## Structure and verification

The engine follows NET-ADR-009: a module is a single file until a child earns a place,
the shared model of a subsystem stays in its trunk, a child exposes its specialization
upward with `pub(super)` and the trunk re-exports outward with `pub use`, and the crate
root declares every module private and names the public surface in one list
(`crates/ton-net-cell/src/lib.rs`). Node-scale breadth and the optional features are
isolated files or feature-gated modules a lighter build does not compile.

Verification for the milestone:

- The two oracles above are the correctness gate, run against captured mainnet bytes in
  `crates/ton-net-cell/tests/cell/`.
- The in-crate property tests over generated trees hold the round trip, the hashing, and
  the encoding-uniqueness rule that the tests over parsed bags cannot construct
  (`crates/ton-net-cell/src/proptests.rs`).
- The performance bar is measured by the criterion benchmarks
  (`crates/ton-net-cell/benches/cells.rs`, `dict.rs`), and each scale path is held to the
  in-memory path's identities while bounding memory.
- The optional paths hold at build time: `compress` and `json` are off the default build,
  and each is the only thing that pulls its dependency (NET-ADR-010, NET-ADR-004).
- `cargo deny check` admits the dependency tree under the existing policy with no new
  exception.
