# ton-net v0.3.0 implementation plan

The trust anchor. v0.2.0 proved an account read against a masterchain block hash the
caller supplied, which reduced trust to one input without removing it. v0.3.0
produces that input: the client walks a chain of key blocks from a single pinned
block, checking a validator-set signature at every step, and arrives at a current
masterchain block it has proven for itself.

After this milestone a read is trust-minimized end to end. The client is handed one
block identity, in the network config it was already given, and everything else it
believes it derived. A liteserver can still stall, withhold, or disconnect, but it
cannot make the client accept a balance, a block, or a state that the validators did
not sign.

This is the release model's line (NET-ADR-004): v0.1.0 and v0.2.0 reached a registry
only as marked pre-releases because a read was the server's word or rested on a hash
the caller had to source. v0.3.0 is the **first ordinary registry release**.

The riskiest pieces are pinned against the reference implementation and against
mainnet before the library build starts, the same spike-first discipline v0.1.0 and
v0.2.0 followed.

## Delivers

From Rust and from Node: connect to TON and read an account that is proven against a
block the client established for itself, with nothing supplied by the caller but the
network config. The block the read is anchored at can be inspected, saved, and handed
back on the next run so the anchor is established once rather than every time.

What a consumer can do after v0.3.0:

- Read an account and get a `Verified<Account>` without sourcing a block hash from
  anywhere.
- See which block the read was proven against, and persist that block so a later run
  starts from it.
- Start from a block of their own choosing instead of the one the config names.
- See a chain that skips a block, misdeclares its direction, carries too few
  signatures, carries signatures from validators outside the set, or ends at a block
  older than the freshness bound, each refused with a distinct error.

What a consumer cannot yet do, and the release says so: resolve an address through
the DHT (v0.5.0), or run a get-method such as a jetton balance or a DNS lookup,
because that needs the TVM (v0.7.0, NET-ADR-005). A verified TON balance is a proven
read; a jetton balance is not proven until the TVM lands.

## Scope

In scope:

1. **Block proof TL** in `ton-net-tl`: `liteServer.getBlockProof` and the reply types
   it names, `partialBlockProof`, `blockLinkForward`, `blockLinkBack`, both arms of the
   boxed `SignatureSet` union and `liteServer.signature`; and the two messages a
   signature covers, `ton.blockId` and the Simplex vote with its `consensus.dataToSign`
   envelope.
2. **Block header decode** in `ton-net-block`: the `BlockInfo` fields a link check
   reads, and the key block's own configuration through `McBlockExtra`, the
   configuration dictionary, and the validator set in parameter 34.
3. **Signature verification** in `ton-net-block`: ed25519 verification, the validator
   short id, the weight rule that decides whether a signature set carries a block, and
   **both signed forms**, because mainnet moved from one to the other partway through
   the range a sync has to cross and a client that speaks only the older form stops
   three quarters of the way to the head.
4. **Link and chain verification** in `ton-net-block`: a pure function over the bytes
   of one link and the block it starts from, and the ordering rules that make a
   sequence of links a chain.
5. **The sync engine** in the `ton-net` facade: the loop that walks from the anchor to
   the head, the bounds that keep a hostile server from driving it, the freshness
   check, and the anchor a caller can read back and pass in.
6. **Config**: parse `validator.init_block` from `global.config.json`, and re-bundle
   the mainnet snapshot with the section it needs.
7. **Node binding** update: sync across the FFI boundary, the anchor in and out, and
   a verified read that needs no caller-supplied block.
8. **First ordinary registry release**: version `0.3.0`, the Rust crates on crates.io
   and the Node binding on npm, with the API reviewed before it becomes a promise.

Out of scope for v0.3.0, each with its own later milestone: the **DHT** and
ADNL-over-UDP (v0.5.0); the **TVM** and get-method execution (v0.7.0); transaction
and block-list proofs; and syncing shard blocks, since an account read derives its
shard block from the masterchain state rather than syncing to it.

Deliberately not built, with the reasoning recorded below rather than left implicit:
the validator list shuffle and the `gen_validator_list_hash_short` cross-check, which
the [validator set](#the-validator-set) section shows cannot change which signatures
are accepted; and backward links, which an anchor that is always a key block never
needs.

## Workspace layout

No crate joins the workspace. The split v0.2.0 established already has the two seams
this milestone needs.

```
crates/
  ton-net-tl/       + getBlockProof, the block links, ton.blockId
  ton-net-cell/     unchanged
  ton-net-adnl/     unchanged
  ton-net-block/    + block header decode, the validator set,
                      signature verification, link and chain verification
  ton-net-lite/     + the getBlockProof call
  ton-net/          + the sync engine, the anchor, config init_block
spikes/
  sync-feasibility/ NEW: the proving spike for this milestone
```

Verifying a link is a pure function of bytes, so it belongs beside the proof engine in
`ton-net-block`. Choosing which block to ask about next is I/O policy, so it belongs
in the facade. That division is the same one NET-ADR-001 fixes for the rest of the
stack, and it is what lets a captured proof chain become a hermetic test.

## The trust model: from a caller's hash to a pinned block

v0.2.0 named three things: a reported value, a verified value, and the anchor a
verified value was checked against. v0.3.0 changes only where the anchor comes from.

- **Reported** (`ServerReported<T>`, unchanged): the liteserver's answer, decoded but
  not proof-checked.
- **Verified** (`Verified<T>`, unchanged): an answer proven against an anchor.
- **The anchor**: no longer an input the caller has to source. The client derives it
  from a block identity it was given once, by walking key blocks and checking that
  each one carries signatures from the validator set the one before it named.

The pinned block is the network config's `init_block`. That matters more than it
looks. The anchor is not a constant this library invents and asks to be believed; it
is a field of the file that already decides which network the client is on, published
and served by the same source the liteserver list comes from. A caller who does not
want to trust that file can pass their own starting block instead. Either way the
trusted input is one block identity, it is visible, and it is replaceable.

Everything after that block is derived. A key block names its validator set in its own
body. The next key block is accepted only if enough of that set, by weight, signed it.
Repeating that from the pinned block to today is the sync, and its output is a current
masterchain block that the client can hand to the v0.2.0 proof engine unchanged.

That the proof engine is unchanged is the point. v0.2.0 built a machine that turns one
trusted block hash into a proven account. v0.3.0 does not modify that machine; it stops
asking the caller to feed it.

## What is still trusted after v0.3.0

Stated plainly, because a trust-minimization claim is only worth its exclusions.

- **The pinned block.** One block identity, from the config the caller chose. Anyone
  can check it against another source, and a caller can substitute their own.
- **The validators themselves.** If more than two thirds of the masterchain validator
  set by weight sign a block, this client accepts it. That is TON's security model,
  and no light client can be stronger than the consensus it reads.
- **Liveness, not soundness.** A liteserver can refuse to answer, answer slowly, or
  serve a head that is real but old. A stale head is caught only by comparing the
  block's generation time against the local clock, so a client with a wrong clock has
  a weaker freshness guarantee. Nothing here lets a server forge; it lets a server
  stall.
- **A single server's view.** This release talks to one liteserver at a time. Talking
  to several and requiring them to agree would catch a server that withholds, and is
  deliberately not in this milestone.
- **This implementation.** The code is open and the tests are in the repository; that
  is the offer, not a claim of correctness.

## The block proof chain

`liteServer.getBlockProof mode:# known_block:tonNode.blockIdExt
target_block:mode.0?tonNode.blockIdExt` asks a server to connect a block the client
already trusts to one it wants. The reply is a `partialBlockProof` carrying `from`,
`to`, a `complete` flag, and a vector of links. The server chooses the route; the
client validates every step of it and believes nothing about the route itself.

A **forward link** goes from a key block to a later block. It carries a Merkle proof
of the source block, a proof of the destination block's header, a proof of the source
key block's configuration, and a set of signatures. Checking it means: the source proof
roots at the source block's known root hash; the configuration proof yields the
validator set the source key block named; the destination header proof roots at the
destination's stated root hash; and the signatures over the destination's identity
carry more than two thirds of that set's weight.

A **backward link** goes from a block back to an earlier one, and cannot use
signatures, because a block is not signed by validators who came later. It carries a
proof of the source block's state and shows that the destination is recorded in that
state's list of previous masterchain blocks. This release does not implement backward
links; see [the sync engine](#the-sync-engine) for why an honest chain never contains
one, and what happens if a server sends one anyway.

The structural rules a link must satisfy are taken from the reference implementation
rather than inferred, and each one is a check in the code:

- Both ends are masterchain blocks.
- The two ends have different sequence numbers, and the direction the link declares
  agrees with which end is higher.
- A proof of the source block is present, and a proof of the destination header is
  present.
- A forward link carries signatures; a backward link carries a state proof.
- The destination header's key-block bit equals the link's `to_key_block` flag, so a
  server cannot present an ordinary block as a key block and have the chain continue
  from a body that carries no configuration.
- The destination header's generation time is read and carried out, because that is
  what the freshness check consumes.

The rules that make a sequence of links a chain sit above the individual link check:
the first link starts at the block the client trusts, each link starts where the last
one ended, the last link ends at the target when `complete` is set, and every
intermediate destination is a key block.

## The validator set

This is the part where a light client most easily goes subtly wrong, so it is written
from the reference implementation and recorded here rather than left to the code.

A key block carries the network configuration in its own body, which is exactly why
key blocks are the waypoints of a sync. Parameter 34 holds the current validator set:
a count of `total` validators, a count of `main`, a total weight, and a dictionary of
descriptors keyed by index, each holding a public key and a weight.

The masterchain validator set for a block is **the first `min(main, total)` entries of
that list, with their weights unchanged**. The reference implementation may then
permute those entries with a seeded generator, depending on a flag in the catchain
configuration, but the permutation is over exactly those entries: it reorders them and
changes neither which keys are in the set nor what each one weighs.

That has a direct consequence, and it is the reason this milestone is smaller than it
first appears. Signature checking asks two questions of the set: is this signer in it,
and what does that signer weigh. Neither answer depends on order. **The shuffle is
therefore not implemented**, and neither is the `gen_validator_list_hash_short`
cross-check in the destination header, which is the only check that does depend on
order.

Skipping a cross-check deserves an argument, not a shrug. The argument is that the
check is redundant against a set derived correctly, and that deriving it incorrectly
fails loudly rather than silently, provided one rule is followed: **the set of accepted
signers and the total weight in the denominator are read from the same set, never from
different ones.** With that rule, taking too many validators makes the threshold
unreachable and taking too few discards honest signatures, and either way live mainnet
blocks stop verifying. A chain that syncs across hundreds of real key blocks is
therefore evidence the derivation is right, in a way that no amount of reading is.

That evidence exists. Walking the whole chain checks 107475 signatures against sets
derived this way, and not one signer falls outside the head-of-list subset. Mainnet runs
about 100 main out of about 380 total, so a wrong subset would have left roughly three
quarters of signers unrecognised rather than none. The list-hash check is recorded as
hardening for v0.9.0, where the seeded generator can be built and pinned against
captured headers.

**One check that looks obvious and must not be made.** A validator set names the window
it is valid for, and requiring a destination block's `gen_utime` to fall inside the
source set's window is the natural-looking guard. It is wrong. The key block that
announces a rotation is stamped one second past the outgoing set's `utime_until` and is
signed by that outgoing set, so the guard refuses the first link of every rotation, and
a full sync crosses 607 of them. The set to use is the one the source key block names,
and the window is not a check.

## Signature verification

A signature in a set is a validator's short id and 64 bytes. The short id is the
SHA-256 of the public key in its TL `pub.ed25519` form, which is the same computation
the ADNL handshake already performs for a server key, so the rule is not new to this
codebase.

**There are two signed forms, and a sync has to speak both.** Mainnet moved from one to
the other at masterchain block 59379986, part way through the range between the config's
pinned block and today, so this is not a compatibility nicety: a client that speaks only
the older form walks three quarters of the way and stops.

**The older form** signs the destination block's identity directly: a four-byte
constructor id for `ton.blockId`, then the block's root hash and file hash, 68 bytes in
total. The file hash is worth a note, because it is the one field in a block identity
that no Merkle proof can establish, being a hash of the serialized block file rather
than of the cell tree. The signature is what makes it trustworthy, since the validators
signed the root hash and the file hash together. This is why a link's destination is
believed only after its signatures check and not after its header proof checks.

**The newer form** comes from TON's Simplex consensus and signs a vote rather than a
block. The set carries a session id, a slot, and the serialized hash-data of the block
candidate, and the message is built from them:

```
consensus.dataToSign {
    session_id,
    data = consensus.simplex.finalizeVote {
        id = consensus.candidateId { slot, hash = sha256(candidate) }
    }
}
```

Two things in that are load-bearing. The vote is a **finalize** vote, which is what
commits a block, rather than the notarize vote that only certifies a candidate. And the
vote is never signed on its own: it is wrapped in a `consensus.dataToSign` beside the
session id, so a signature raised in one consensus session cannot be replayed into
another. The candidate bytes are hashed whole and never need decoding, though decoding
one confirms it names the same block the link does.

Every constructor id in both forms is derived from its schema line the way the rest of
the codebase derives them, not written down as a literal.

A set carries the block when the total weight of the valid signatures from distinct
validators in the set is more than two thirds of the set's total weight. Three details
are checks, not assumptions: a signer not in the set contributes nothing rather than
failing the whole set, a validator counted twice contributes once, and **the comparison
is integer arithmetic**, `weight * 3 > total_weight * 2`, with no rounding and no
floating point anywhere near it.

That last point is not fastidiousness. The thinnest link on the real chain from the
pinned block to today carries 66.6712% of its set's weight against a two-thirds
threshold: **0.0046 percentage points of headroom**, about a two-hundredth of a single
validator's weight. A comparison done in `f64` at that margin is a coin toss.

The same measurement decides the acceptance rule for an individual signature, and it
overturns the obvious answer. Ed25519 verifiers famously disagree on edge cases, and the
tempting position is that stricter is safer because a stricter rule can only refuse. At
a margin of 0.0046 points, refusing a single signature the validators accepted stalls
the sync outright. So the target is not maximum strictness but **the rule the network
itself uses**: libsodium's `crypto_sign_verify_detached`, which is what validators sign
and verify with. Canonical `A`, `R` and `S`, neither `A` nor `R` of small order, and the
uncofactored equation. This client aims to accept exactly that set, no more and no less,
and the rule is pinned by vectors and by a differential test rather than asserted.

## The sync engine

The loop is short, because the protocol already puts the routing on the server.

1. Read the head the server reports.
2. Ask for a proof from the anchor to that head.
3. Validate every link and the chain rules, advancing the anchor as each link checks.
4. If the reply was not complete, repeat from the new anchor, requiring the sequence
   number to have advanced.
5. Check the final block's generation time against the local clock.

The anchor the client keeps between calls is always **a key block**, never the head it
last read. That single rule is what removes backward links from the picture: a
backward link exists so a client whose known block is not a key block can reach the
last key block before it, and a client whose known block is always a key block never
needs one. A server that sends a backward link anyway gets a distinct error naming it,
rather than a skipped step or a misread one.

The head a read is performed at is proven for that read and then discarded. Keeping it
would mean re-syncing from a non-key block on the next call, which is the situation the
rule above avoids.

A caller who holds a proven key block from an earlier run passes it back at connect
time, and the sync walks forward from there instead of from the config's pinned block.
That block is a root of trust: anything that can write to where a caller keeps it can
choose what this client believes. The library therefore stores nothing, picks no
location, and writes no file. It takes a block identity in and hands one back, and
where that lives is a decision the caller's own threat model makes.

## Refusing a hostile server

The sync loop is the first place in this library where a server decides how much work
the client does, so the bounds are part of the design rather than a later hardening
pass.

- A cap on links per reply, and on links across a whole sync, so a chain cannot run
  without end.
- A cap on the bytes of a single proof, on top of the cell engine's existing cell and
  depth bounds.
- A cap on signatures in a set, checked before any verification runs, so a set cannot
  be a compute amplifier.
- Strict progress: an incomplete reply that does not raise the anchor's sequence number
  ends the sync with an error instead of looping.
- A freshness bound on the final block's generation time, with the bound exposed so a
  caller whose clock or latency demands something different can say so.

Every one of these ends a sync with a named error. None of them relaxes a check to make
a sync succeed.

## What a cold sync costs

Measured on 2026-07-21 by walking it, so these are observations rather than estimates.

The mainnet config's `init_block` is at masterchain sequence number 46894135, and the
head that day was 81057314: 460 days and 34.2 million masterchain blocks apart. Key
blocks arrive twice per validator round, at 8192 seconds and then 57344 seconds within
each 65536-second round. A full cold sync over that span:

| | |
|---|---|
| round trips | 78, at 16 links per reply |
| links | 1242 key blocks |
| signatures verified | 107475 |
| validator-set rotations crossed | 607 |
| downloaded | 52.6 MB, 43.4 kB per link |
| elapsed | 154 s, of which 148 s waiting on the network |

The cost is dominated by the configuration proof each link carries, about 36 kB of it,
because that proof has to expose a validator set of several hundred entries. It is not
compressible by asking differently: it is what a key block has to hand over for the next
one to be checkable.

A warm sync from a key block saved the previous day is two or three links.

The distance grows by about 800 key blocks a year only while the pinned block stands
still. It is published in the config, and a caller who fetches a current config gets a
current block, so the walk is bounded by how stale the config is rather than by how old
the network is. That makes refreshing the bundled snapshot part of cutting a release
rather than housekeeping, and it is the reason the snapshot's date belongs next to it.

Three things follow, and they shape the API more than the cryptography does. The cost
is paid at connect, where a caller expects setup to happen, rather than inside a read.
The anchor is handed back so the cost is paid once rather than per process. And the
measured figures go in the README, because a library whose first call is slow should
say so in advance rather than be discovered.

The design's largest unknown is answered by the same run: a public liteserver does serve
a proof chain from a block 460 days old. Key blocks are retained where ordinary blocks
are pruned, which is what makes any of this work, and it is now confirmed rather than
assumed.

## Public API

### `Client`, from the facade

```rust
impl Client {
    /// Connects to the network and establishes the block the client trusts.
    ///
    /// The anchor starts at the `init_block` in `config` and is walked forward to a
    /// current block, checking a validator signature set at every key block. On a
    /// first run this walks every key block since the config was published, which
    /// costs bytes and time; [`connect_from`](Self::connect_from) starts from a block
    /// a caller already proved.
    ///
    /// # Errors
    ///
    /// Returns [`Error::Config`] if the config names no init block, [`Error::Sync`] if
    /// the proof chain does not check out, [`Error::Stale`] if the block the server
    /// leads to is older than the freshness bound, or the transport errors.
    pub async fn connect(config: &Config) -> Result<Client, Error> { /* ... */ }

    /// Connects and starts the walk from a block the caller already trusts.
    ///
    /// `anchor` must be a masterchain key block. It is the client's root of trust:
    /// everything the client goes on to believe is derived from it, so it is worth
    /// exactly what the storage it came from is worth.
    ///
    /// # Errors
    ///
    /// As [`connect`](Self::connect), and [`Error::Sync`] if `anchor` is not a
    /// masterchain block the server can start a proof from.
    pub async fn connect_from(config: &Config, anchor: &BlockIdExt) -> Result<Client, Error> { /* ... */ }

    /// The key block the client's trust currently rests on.
    ///
    /// Saving this and passing it to [`connect_from`](Self::connect_from) on a later
    /// run makes that run's sync short. It is a public block identity and holds no
    /// secret, but whatever can write to where it is kept can choose what a later
    /// client believes.
    #[must_use]
    pub fn anchor(&self) -> &BlockIdExt { /* ... */ }

    /// Walks the anchor forward to the network's current head.
    ///
    /// Called for the caller by every verified read, so an explicit call is only
    /// needed to control when the cost is paid.
    ///
    /// # Errors
    ///
    /// Returns [`Error::Sync`] or [`Error::Stale`] as the walk fails.
    pub async fn sync(&mut self) -> Result<&BlockIdExt, Error> { /* ... */ }

    /// Reads an account and proves it against a block the client established itself.
    ///
    /// Syncs if the anchor is behind, reads the account at the proven block, and
    /// checks the proof. Nothing in the result rests on a hash the caller supplied.
    ///
    /// # Errors
    ///
    /// Returns [`Error::Sync`], [`Error::Stale`], [`Error::Proof`], [`Error::Cell`],
    /// or the transport errors.
    pub async fn account(&mut self, address: &Address) -> Result<Verified<Account>, Error> { /* ... */ }

    /// Reads an account and proves it against a block the caller supplies.
    ///
    /// The v0.2.0 read, kept for a caller who has an anchor of their own and does not
    /// want the client to choose one.
    pub async fn account_at(
        &mut self,
        address: &Address,
        trusted: &BlockIdExt,
    ) -> Result<Verified<Account>, Error> { /* ... */ }

    /// Reads an account without checking a proof.
    ///
    /// The value is the server's word. The name says so, because after this release a
    /// read that is not proven is the exception.
    pub async fn account_reported(
        &mut self,
        address: &Address,
    ) -> Result<ServerReported<Account>, Error> { /* ... */ }
}
```

`account` and `account_verified` change meaning and name from v0.2.0. The default read
becomes the proven one, and the unproven read is reachable only by asking for it by
name. This is the last release before the API becomes a published promise, so it is
the right and cheapest moment to make the safe path the one a caller lands on without
choosing.

### Link verification, from `ton-net-block`

```rust
/// One step of a block proof chain, as a liteserver sent it.
#[derive(Debug, Clone, Copy)]
pub struct BlockLink<'a> { /* ... */ }

/// Checks a forward link and returns what the destination block header says.

pub fn verify_forward_link(link: &BlockLink<'_>) -> Result<BlockHeader, BlockError> { /* ... */ }

    /// The block's sequence number.

    /// The sequence number of the last key block before this one.
```

## Error taxonomy

The facade keeps its single `#[non_exhaustive]` `Error`, with two variants added.

```rust
#[derive(Debug, thiserror::Error)]
#[non_exhaustive]
pub enum Error {
    // ... the v0.1.0 and v0.2.0 variants, unchanged ...

    /// A block proof chain did not check out.
    #[error("sync: {0}")]
    Sync(String),

    /// The newest block the server could prove is older than the freshness bound.
    #[error("the proven head is {age_seconds}s old, past the {limit_seconds}s bound")]
    Stale {
        /// How far behind the local clock the proven head is.
        age_seconds: u64,
        /// The bound that was exceeded.
        limit_seconds: u64,
    },
}
```

`Sync` and `Proof` stay separate because they fail differently. A `Proof` failure means
a server's answer did not match a block; a `Sync` failure means the client never
established a block to match against, and a caller retrying against another server is a
reasonable response to one and not the other. `Stale` is separate from both because it
is the one failure where the server may be honest and merely behind.

Three invariants hold, all testable:

- **A chain that does not connect is refused.** A missing link, a repeated block, a
  reversed direction, or a chain that stops short of its target yields `Error::Sync`.
- **A set that does not carry the block is refused.** Signatures below the weight
  threshold, from validators outside the set, duplicated, or over a different block
  yield `Error::Sync`, never a shortened chain that quietly stops early.
- **A sync terminates.** Any sequence of replies, including an adversarial one, ends in
  an anchor or an error within the bounds, never an unbounded loop or allocation.

## The Node binding surface

The trust store does not cross the boundary; a block identity does. That keeps the
binding free of storage policy and gives every later binding the same shape.

```ts
export class TonClient {
  /** Connects and establishes the trusted block. The first run walks every key
   *  block since the config's init block, which takes time and bandwidth. */
  static connect(config: Config): Promise<TonClient>

  /** Connects starting from a key block proven on an earlier run. */
  static connectFrom(config: Config, anchor: BlockId): Promise<TonClient>

  /** The key block this client's trust rests on. Save it, pass it to connectFrom. */
  anchor(): BlockId

  /** Walks the anchor forward to the current head. */
  sync(): Promise<BlockId>

  /** Reads an account, proven against a block the client established itself. */
  account(address: string): Promise<Verified<Account>>

  /** Reads an account, proven against a block the caller supplies. */
  accountAt(address: string, trusted: BlockId): Promise<Verified<Account>>

  /** Reads an account without checking a proof. The server's word. */
  accountReported(address: string): Promise<ServerReported<Account>>
}
```

`BlockId` already crosses in both directions as of v0.2.0, so persisting an anchor from
JavaScript is saving an object with two buffers and reading it back.

## Testing and conformance

- **Hermetic tests**, no network, on every commit:
  - A captured real proof chain spanning a validator-set rotation, replayed and
    verified end to end against the block it starts from.
  - A tamper corpus over that chain: a flipped signature byte, a signature moved to
    another link, a validator id swapped for one outside the set, a weight altered in
    the configuration proof, a link removed, two links transposed, a direction flag
    inverted, a key-block flag inverted, a chain truncated before its target. Each is
    refused, and the test asserts which error, not merely that one occurred.
  - The weight rule at its boundary: a set at exactly two thirds is refused and a set
    one unit above it is accepted, over synthetic sets, because the boundary is where
    an off-by-one hides.
  - Ed25519 acceptance vectors, including the published edge cases for small-order
    keys, non-canonical scalars, and torsion components, so the acceptance rule is
    pinned rather than inherited.
  - A hostile-bytes test over the link and chain decoders, as `ton-net-cell` and
    `ton-net-tl` already have.
- **Network tests**, `#[ignore]`, in the dedicated network job:
  - A cold sync from the config's init block to the head, asserting the walk crosses at
    least one validator-set rotation and reporting the links, bytes, and elapsed time.
  - A warm sync from the key block the cold sync ended at, asserting it is short.
  - A verified account read with no caller-supplied anchor, with the balance matched
    against an independent public source at a block both agree on.
  - A sync started from a block that is real but not on the chain, refused.
- **The Node gate test**: connect, read the anchor, read an account verified against
  it, reconnect from the saved anchor and assert the second sync is short, and assert
  a tampered anchor is refused.

The cold-sync test is the milestone's gate and is the one test that cannot be made
fast. It runs in the network job on a schedule rather than on every commit, and its
measured cost is recorded so a regression in the number is visible.

Two test-design rules carry over from v0.2.0, where they caught real problems. An
assertion that can only pass is not a test: the rotation crossing is asserted by
reading the validator set on both sides of it and requiring the sets to differ, not by
assuming a long enough walk must have crossed one. And an independent source is only
independent if it is not the server being questioned, which for the balance match means
a public API rather than a second call to the same liteserver.

## Dependencies

**This milestone conflicts with NET-ADR-003 and the conflict has to be resolved before
the code is written, not around it.** The ADR requires a single copy of each crypto
crate in the tree and rejects `ed25519-dalek` by name, on the grounds that the ADNL key
conversion needs only `curve25519-dalek`. v0.3.0 is the first release that verifies an
ed25519 signature, so the question reopens on new facts:

- `ed25519-dalek` 3.0 depends on `curve25519-dalek` 5, which is the version already in
  the tree, but also on `sha2` 0.11, which duplicates `sha2`, `digest`, `crypto-common`
  and `block-buffer` against tl-proto's 0.10 generation. This is the exact duplication
  the ADR was written to prevent.
- `ed25519-dalek` 2.2 depends on `curve25519-dalek` 4, which duplicates the curve crate
  instead.

Neither version can be added without breaking the ADR's single-copy rule. The proposal
is therefore to verify signatures on `curve25519-dalek` 5 and `sha2` 0.10, both already
present, which adds **no new dependency at all**. The operations are the ones the crate
exposes for this purpose: a canonical scalar decode, a wide reduction of the SHA-512
digest, and the double-scalar multiplication against the basepoint.

This is a deliberate step toward a primitive rather than away from one, and it is the
same call the ADR already made once for the ADNL handshake, so it is an amendment in
the ADR's own direction rather than a reversal. It is not free of risk: signature
verification is a place to be careful, and "we wrote our own" is a fair question for any
library on a trust path. Three things answer it, and all three are gates rather than
intentions:

- The acceptance rule is pinned by the published ed25519 edge-case vectors, which are
  precisely the cases implementations disagree on.
- `ed25519-dalek` is added as a **development dependency** and the two are compared
  against each other over random and adversarial inputs. A development dependency does
  not ship, does not enter a consumer's tree, and does not duplicate anything a
  consumer builds.
- The rule is exercised against roughly a hundred thousand real mainnet validator
  signatures on the cold-sync run, which is a larger corpus than any vector set.

If the amendment is not accepted, the fallback is `ed25519-dalek` 3.0 with the `sha2`
duplication recorded as a known cost. That is a defensible outcome too, and the decision
is recorded either way in a new ADR rather than settled inside a commit.

No other dependency changes. `cargo tree -d` stays clean for what ships.

## Sequence

Ordered steps, each ending in something checkable. A step does not start until the one
before it is green.

0. **Feasibility spike** (`spikes/sync-feasibility`). **Done**, and its findings are
   already folded into the sections above. It walked the whole chain from the config's
   init block to the head, checking every link, and settled: that a liteserver serves a
   chain from a block 460 days old; the cold-sync cost as a measured number; that the
   masterchain set is the head of the parameter-34 list, over 107475 signatures; that no
   backward link appears in 1242; that ed25519 verification needs no new dependency;
   that the older signed form is `ton.blockId` and not the approval form; and that a
   second signed form exists, is now the only one in use, and had to be reverse-derived
   from real signatures. Its README records the two layout traps it caught. What remains
   for step 4 is a trimmed hermetic corpus: the captures are whole 700 kB replies, and
   the fixtures the library needs are a handful of links, one of each signed form.
1. **The ADRs.** Written: [NET-ADR-006](../adr/NET-ADR-006-trust-anchor.md) records the
   trust anchor and everything that follows from it, and
   [NET-ADR-007](../adr/NET-ADR-007-signature-verification.md) records the signature
   verification decision and amends NET-ADR-003. Both are `proposed`. Exit: both
   accepted, because there is no implementation without them.
2. **`ton-net-tl` and `ton-net-lite`.** `getBlockProof`, the reply types, the boxed
   signature-set union with an unknown constructor refused by name, and `ton.blockId`.
   Exit: a captured reply round-trips through the codec byte for byte, and an unknown
   constructor errors instead of misreading the fields that follow it.
3. **`ton-net-block`, the header and the set.** `BlockInfo` for the fields a link check
   reads; the key block's configuration through `McBlockExtra`; parameter 34 and the
   validator set; the validator short id. Exit: a captured key block yields the exact
   validator set, count and total weight that an independent source reports for that
   round.
4. **`ton-net-block`, signatures and links.** Ed25519 verification with its vectors and
   the differential test; the weight rule; forward-link verification; the chain rules.
   Exit: the captured chain from step 0 verifies end to end, hermetically, and every
   member of the tamper corpus is refused with the expected error.
5. **The sync engine.** The loop, the bounds, the freshness check, the anchor in and
   out, and `Config::init_block` with the mainnet snapshot re-bundled. Exit: a cold sync
   from the config's init block reaches the head on mainnet across a validator-set
   rotation, and a warm sync from the resulting key block is short.
6. **The facade read path.** `account` returns `Verified<Account>` with no caller
   anchor; `account_at` and `account_reported` take the explicit roles. Exit: the
   network test reads a verified account with nothing supplied but the config, and the
   balance matches an independent source.
7. **`bindings/node`.** Sync across the boundary, the anchor in and out, and the
   renamed read methods. Exit: the Node gate test connects, verifies an account,
   reconnects from the saved anchor, and shows the second sync is short.
8. **Release.** Review the API before it becomes a promise: every public item
   documented, `#[non_exhaustive]` where a type may grow, and nothing exposed that is
   not meant to be supported. Tag `v0.3.0`. Publish the Rust crates to crates.io bottom
   up and the Node binding to npm as `0.3.0`, the first ordinary release (NET-ADR-004),
   with the measured sync cost in the README.

## Decisions made in this plan

- **The anchor is the config's init block, not a constant this library ships.** The
  network config already decides which network a client is on and which servers it
  talks to; the block it trusts belongs in the same file, from the same publisher, where
  a caller can see and replace it.
- **The anchor is always a key block.** It makes every step of a sync a forward link,
  which removes backward links and the masterchain state parsing they need from this
  milestone, and it means a saved anchor is always a valid starting point.
- **The library stores nothing.** A saved anchor is the root of trust, so where it lives
  is a security decision the caller's threat model makes. A block identity goes in and
  comes back out; no file is written and no location is chosen. A storage trait was
  considered and deferred, because it is policy this release has no evidence for and
  because it would not cross a binding boundary cleanly.
- **The shuffle and the list-hash check are not built.** The masterchain set is the
  first `min(main, total)` entries either way, so the permutation cannot change which
  signatures are accepted or what they weigh. Deriving the accept-set and the weight
  denominator from the same set makes a wrong derivation fail on live data, which is the
  check that matters. The list-hash cross-check is hardening for v0.9.0.
- **Ed25519 verification is built on the curve crate already in the tree.** Adding
  `ed25519-dalek` duplicates either the curve or the whole digest generation, both of
  which NET-ADR-003 forbids. The audited implementation is used as a development
  dependency for differential testing, which keeps the cross-check without shipping the
  duplication.
- **Strict signature acceptance.** Strictness can only refuse, never wrongly accept, and
  the signatures it refuses are ones honest validators do not produce. The liveness edge
  is named rather than hidden.
- **The default read becomes the proven one.** After sync exists, a default that returns
  the server's unchecked word is a trap. `account` proves, `account_reported` does not
  and says so. Renaming costs nothing now and would cost consumers later.
- **Connect pays the sync.** A client that has not established an anchor cannot do its
  job, and connect is where a caller expects setup cost. A read that silently spends
  tens of megabytes is worse than a connect that does.
- **Freshness is a clock comparison, with the bound exposed.** No proof carries the time
  it was served. A stale head is the honest limit of a single-server light client, so the
  check is stated, defaulted, and left adjustable rather than presented as a guarantee.

## Open questions

- **Closed by step 0.** A liteserver does serve a chain from a block 460 days old. The
  older signed form is the commit form, not the approval form. The masterchain set is the
  head of the parameter-34 list. No backward link appears in an honest chain. Each is
  recorded above with the evidence rather than left as an assumption.
- **Whether the cold sync is acceptable as a default.** Now a measured 52.6 MB and 154
  seconds, which is a real cost to put in front of a first-time caller. The options are
  to leave it and state it, to make connect lazy so the cost lands on the first read
  instead, or to ship a more recent checkpoint, which reintroduces a trust input this
  plan deliberately removed and would need its own ADR. Leaving it and stating it is the
  proposal, on the grounds that the anchor is saved and the cost is paid once.
- **Whether the older signed form is worth keeping at all.** It is needed only to cross
  the stretch between the config's pinned block and block 59379986. A caller starting
  from a recent anchor never meets it. Keeping it is what makes the config's own pinned
  block usable, so it stays, but it is the first thing to drop if the pinned block moves
  past the changeover.
- **Whether a third signed form is coming.** One arrived inside the span this milestone
  has to cross, which is evidence the format is not settled. The codec refuses an
  unknown constructor by name rather than misreading it, so a future form is a clean
  failure rather than a wrong answer; treating that failure as a routine event, with a
  clear message telling a caller the library is behind the network, is the honest
  posture.
- **Whether the registry names should be claimed before v0.3.0 ships.** All six crate
  names and both npm names were unregistered on 2026-07-21. They are not reserved by the
  repository existing, and a first ordinary release is the moment they stop being
  available to anyone else. Publishing the current pre-release to hold them is a
  decision this plan raises and does not make.

## Definition of done

v0.3.0 is done when:

- From Rust and from Node, an account read returns a verified account with nothing
  supplied by the caller but the network config, and the block it was proven against can
  be read back.
- A cold sync from the config's init block reaches the current head on mainnet, across
  at least one validator-set rotation proven by the sets on either side differing, and a
  warm sync from the resulting key block is short.
- Both signed forms verify, proven by a hermetic fixture of each and by a cold sync that
  necessarily crosses the changeover, and an unrecognised third form fails with a message
  naming it rather than being misread.
- The weight comparison is integer arithmetic, tested at the boundary, and the signature
  acceptance rule matches libsodium's under differential test. The thinnest real link has
  0.0046 percentage points of headroom, so neither is a formality.
- The tamper corpus is refused, each member with the error the test names: a forged
  signature, a signature from outside the set, a duplicated signer, an altered weight, a
  removed link, transposed links, an inverted direction, an inverted key-block flag, and
  a chain that stops short.
- The weight boundary is tested on both sides, and the ed25519 acceptance rule matches
  the published edge-case vectors and an independent implementation under differential
  test.
- A sync terminates on any sequence of replies, within stated bounds, and a stale head
  is refused rather than accepted.
- The cold-sync cost is measured and published in the README rather than estimated.
- `fmt`, `clippy` with warnings denied, the doc gate, the hermetic suite, the network
  job and the Node gate test are all green.
- The ADR recording the trust anchor and the signature-verification dependency is
  accepted, and NET-ADR-003 is amended or the fallback is taken and recorded.
- `v0.3.0` is tagged, and the crates and the Node binding are published as `0.3.0`, the
  first ordinary release, with the remaining trusted inputs stated in the README.
