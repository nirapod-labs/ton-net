# Proof-engine feasibility spike

A standalone program that proves the v0.2.0 cell hashing against a live TON mainnet
liteserver, building the bag-of-cells parse and the cell hash directly rather than
through a cell library.

v0.2.0's riskiest piece is representation hashing: a Merkle proof is believed only
because recomputing cell hashes reproduces a known root, and the level-mask rule is
subtle (see [docs/plan/v0.2.0.md](../../docs/plan/v0.2.0.md)). This spike de-risks
it before the `ton-net-cell` build, in stages, each checked against a hash the TON
node itself computed, so a wrong hasher cannot pass by agreeing with data this
program also produced. It is the cell-engine counterpart of the ADNL handshake
spike.

It reuses the v0.1.0 facade to fetch a real account and its proof, then hand-rolls
the parse and the hashing.

## Run

```
cargo run
```

It needs outbound TCP to the mainnet liteservers in the bundled config. It reads the
config contract (`-1:5555...5555`), an always-active masterchain account large enough
to be pruned in its state proof.

## Stages, all passing on mainnet

- **Stage 1.** Parse the account state bag of cells, compute the root cell's hash with
  the ordinary-cell rule, and confirm that hash appears in the server's proof as the
  stored hash of a pruned branch. The node put that hash there, so a match proves the
  parse and the ordinary hashing on the unambiguous part of the algorithm.

- **Stage 2.** Recompute each Merkle-proof cell's covered tree at level 0, applying the
  exotic-cell rules, and confirm it against the hash the node stored in that cell. Both
  Merkle cells self-certify.

- **Stage 3.** Follow the chain a verified read depends on: the block proof roots at the
  block hash, the block's state update names the new state root, the state proof roots
  there, and the account hash is the pruned branch it carries.

Result at masterchain seqno 81029319, reading the config contract: the state parses to
2201 ordinary cells; the 1585-byte proof parses to 2 Merkle proofs, 1 Merkle update, 22
pruned branches and 20 ordinary cells; every stage matches. The block hash the chain
roots at was confirmed against a public API independent of the liteserver that served
the proof, so the anchor is not the same server's word.

## Why this is not self-validation

Every check compares against a hash the TON node computed, not one this program
produced. Recomputing a 45-cell SHA-256 chain to land on a block hash an independent
source reports is not something a wrong hashing rule does by accident. This is the
guard the address checksum work in v0.1.0 taught: a fixture generated by the same code
it is meant to test proves nothing.

## The hashing rules it pins

- `d1 = r + 8s + 32l` and `d2 = floor(b/8) + ceil(b/8)`, with the standard
  representation `d1 || d2 || data || ref depths || ref hashes` under SHA-256.
- A pruned branch below its own level answers with the stored hash and depth of the
  subtree it replaced, at `data[2 + 32i]` for hashes and after them for depths. This is
  what makes a pruned tree hash to the root of the full tree.
- A Merkle cell reads its children one level up, and its level is `max(Lvl(ref) - 1, 0)`.
- The level mask is applied per level when forming `d1`, so a cell on a proof path
  hashes at level 0 as the ordinary cell it was before pruning.
